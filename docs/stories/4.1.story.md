# Story 4.1: Persistent Backend Task Management Service

## Status: Draft

## Story

**As a developer,** I need a centralized, persistent service to manage the state and progress of all long-running background tasks.

## Acceptance Criteria (ACs)

* AC1: A `background_tasks` table must be created in the SQLite database to store task information.
* AC2: All new tasks (imports, downloads) must be saved as a record in this table.
* AC3: The service must support parent/child task relationships (for playlist downloads) to handle partial failures gracefully.
* AC4: On application startup, the service must read this table to resume the state of any unfinished tasks, ensuring no progress is lost if the app is closed.

## Tasks / Subtasks

**Backend (`Main Process`)**

* [ ] **Task 1 (AC: 1):** Add the `CREATE TABLE background_tasks` statement to the database schema file (e.g., `src/backend/db/schema.sql`). The schema must match the one defined in the `ArchitectureDocument.md`.
* [ ] **Task 2 (AC: 2):** Create a `TaskRepository.ts` that implements the Repository Pattern for all CRUD (Create, Read, Update, Delete) operations on the `background_tasks` table.
* [ ] **Task 3 (AC: 2, 3):** Create the `TaskManagementService.ts`. This service will use the `TaskRepository` and will be the single point of contact for all task-related operations. It should include methods like `createTask`, `updateTaskProgress`, and `getTasks`.
* [ ] **Task 4 (AC: 3):** Implement the logic for parent/child task relationships within the service, utilizing the `parentId` field in the `BackgroundTask` data model.
* [ ] **Task 5 (AC: 4):** Implement logic that runs on application startup. This logic should query the `background_tasks` table for any tasks that were `IN_PROGRESS` or `QUEUED` and re-add them to the `p-queue` to be processed.
* [ ] **Task 6 (AC: 1):** Implement the IPC mechanism that pushes the list of active tasks to the frontend on the `task:update` channel whenever a task's status changes.

## Dev Technical Guidance

* **Database Schema:** The schema for the `background_tasks` table is explicitly defined in the `ArchitectureDocument.md`. It includes fields like `id`, `type`, `status`, `progress`, `target_id`, and `parent_id`, which are essential for this story.
* **Persistence is Key:** The primary goal of this story is to ensure tasks are durable. If the user closes the app mid-download, the task should be picked back up or marked as failed upon restart. This is handled by reading the database on startup (Task 5).
* **Separation of Concerns:** The `TaskManagementService` should handle the business logic of managing tasks, while the `TaskRepository` should only handle the direct database queries. This aligns with the Repository Pattern defined in the architecture.
* **Parent/Child Logic:** The ability to link tasks is crucial for features like playlist downloads (one parent task for the playlist, many child tasks for the videos). The service must correctly handle creating these relationships and potentially aggregating child progress to the parent.
